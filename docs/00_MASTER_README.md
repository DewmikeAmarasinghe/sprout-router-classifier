# sprout-router-classifier — Master Plan (v4)

## PROJECT OVERVIEW

hSenid Mobile's Sprout chatbot currently routes ALL user messages to `gpt-4o`,
which is expensive. This project builds a fast binary classifier to decide:

  label=0  →  gpt-4o-mini   (pure English, no Sri Lankan context needed)
  label=1  →  gpt-4o        (Singlish, Tanglish, Tamil, Sinhala, SL location,
                              location-implied intent, or complex task)

False negative (predicting 0 when it should be 1) = garbled response to user.
This is unacceptable. Always default to 1 when uncertain.

## WHAT THE MODEL CLASSIFIES — label=1 CONDITIONS

  1. Language is not pure English
     Singlish, Tanglish, Sinhala script, Tamil script.

  2. User mentions a Sri Lankan location
     Caught by rule engine + gazetteer. Model does not need to learn this.

  3. User implies location need without naming one
     "nearest branch to me", "closest outlet in my area"
     Rule engine cannot catch these. Model is explicitly trained on them.

  4. User asks to reply in a non-English language
     "answer me in sinhala", "tamil la kiyanna"

  5. Complex or multi-step task
     Fraud triage, policy comparisons, medical eligibility — gpt-4o-mini degrades.

  6. Ambiguous / very short message
     Safer to route to gpt-4o when intent is unclear.

  label=0 ONLY: pure English, no SL context, straightforward transactional query.

## THE THREE SYSTEM LAYERS

  LAYER 1 — RULE ENGINE      < 0.1ms    Pure Python. No training.
    Catches: Sinhala/Tamil unicode, named SL locations (gazetteer + rapidfuzz).

  LAYER 2 — ML CLASSIFIER    1–30ms     Single trained binary model.
    Catches: everything the rule engine misses.
    Trained entirely on romanized text (English letters).

  LAYER 3 — SAFE DEFAULT      0ms
    confidence < threshold → label=1. Never risk bad UX.

## DATASET STRATEGY

  One dataset. ~62,000 rows. Generated by distilabel.

  WHY 62,000:
    8 signal categories with subtle distinctions. Implied location intent and
    complex-task signals need volume to generalize reliably. Transformers
    benefit from data volume for low-frequency signals.

  TIERED GENERATION:
    Generate by category in batches. Run pipeline after each batch.
    If recall(1) converges before 62k, stop early — distilabel checkpoints.
    62k is the ceiling, not a hard requirement.

  FUTURE PROOFING:
    Folder structure is dataset-agnostic (data/datasets/v1_baseline/).
    Adding a v2 later requires zero refactoring — just run Phase 1 with a new name.
    For now: one dataset, one pipeline run, full focus.

---

## ALL TRAINING DATA IS IN ENGLISH LETTERS

  No Sinhala script or Tamil script in training data.
  Unicode rule catches those 100%. Training budget goes to the hard cases:
  romanized Singlish, Tanglish, implied location, complex tasks.
  A 200-row grounding file verifies the unicode rule (not used for training).

## BASELINES

  Baseline A: Rule engine only (no ML)
  Baseline B: LLM-as-classifier (gpt-4o-mini classifies each prompt — cost reference)
  Trained models must beat both on F1, latency, and cost.

## TOOLS

  distilabel    LLM data generation + evaluation pipeline (Argilla).
  MLflow        Experiment tracking + dataset lineage. No DVC needed.
  Milvus Lite   Semantic deduplication. Single .db file, portable.
  Optuna        HPO. Integrates with MLflow natively.
  Gradio        Launcher UI. Dynamic — reads filesystem for dropdowns.
                No system prompt editing in UI (prompts live in code).
  rapidfuzz     Fuzzy matching for location gazetteer.

## MODULARITY PRINCIPLE

  New classifier   = one line in CLASSIFIER_REGISTRY
  New transformer  = one line in TRANSFORMER_REGISTRY
  New vectorizer   = one line in VECTORIZER_REGISTRY
  Future dataset   = re-run Phase 1 with new --dataset-name, zero refactoring
  Nothing hardcoded anywhere.
